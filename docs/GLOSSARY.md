# Glossary

> **Project-specific terminology for dawson-does-framework.**
>
> This document defines all terms, concepts, and abbreviations used throughout the project. Read this to establish a common vocabulary.

---

## Core Concepts

### Export
**Verb**: The act of creating a new project from a template.

**Usage**: `framework export saas ./my-app`

**What it does**:
- Clones a template to your local machine
- Applies selected integrations
- Generates configuration files
- Initializes git repository
- Creates manifest for tracking

**Related**: Template, Integration, Pull

---

### Pull
**Verb**: The act of downloading a project configuration from the web platform.

**Usage**: `framework pull <token>`

**What it does**:
- Fetches project data from Supabase using a token
- Reconstructs the full project locally
- Applies integrations selected in web UI
- Generates Cursor-specific files (`.cursorrules`, prompts)
- Enables seamless web-to-local workflow

**Related**: Export, Token, Cursor

---

### Template
**Noun**: A pre-built project scaffold with opinionated structure and best practices.

**Available Templates**:
- `saas` - Full-stack SaaS with auth, billing, database
- `blog` - Content-focused blog with SEO
- `dashboard` - Admin panel with data tables and charts
- `landing-page` - High-converting marketing page
- `seo-directory` - SEO-optimized directory site

**File**: `template.json` - Metadata defining template capabilities

**Structure**: See [FILE_STRUCTURE.md](standards/FILE_STRUCTURE.md#template-structure)

**Related**: Export, Integration, Template ID

---

### Template ID
**Noun**: Unique identifier for a template.

**Format**: `kebab-case` (e.g., `saas`, `seo-directory`, `landing-page`)

**Usage**:
```bash
framework export <template-id> <output-dir>
framework templates list
```

**Where used**:
- CLI commands
- `template.json` file
- Directory names under `templates/`

---

### Integration
**Noun**: Optional add-on functionality for a template (auth, payments, AI, etc.).

**Types**:
- `auth` - Authentication (Supabase, Clerk)
- `payments` - Billing (Stripe, Paddle)
- `db` - Database (Supabase, PlanetScale)
- `email` - Email sending (Resend, SendGrid)
- `ai` - LLM integration (OpenAI, Anthropic)
- `analytics` - User tracking (PostHog, Plausible)
- `storage` - File storage (Supabase, S3, Cloudinary)

**Structure**: Each integration lives in `templates/{template}/integrations/{type}/{provider}/`

**Required Files**:
- `integration.json` - Metadata
- `package.json` - Dependencies

**Related**: Provider, Template, Integration Type

---

### Integration Type
**Noun**: Category of functionality (auth, payments, email, etc.).

**Usage**: `--auth supabase` (type = `auth`, provider = `supabase`)

**Format**: Lowercase, no spaces (e.g., `auth`, `payments`, `db`)

**Where defined**: `integration.json` → `type` field

**Related**: Integration, Provider

---

### Provider
**Noun**: Specific service implementation for an integration type.

**Examples**:
- Auth providers: `supabase`, `clerk`
- Payment providers: `stripe`, `paddle`
- Email providers: `resend`, `sendgrid`
- AI providers: `openai`, `anthropic`

**Format**: Lowercase, no spaces

**Structure**: Directory name under `templates/{template}/integrations/{type}/{provider}/`

**Related**: Integration, Integration Type

---

### Token
**Noun**: Short-lived identifier for pulling a project from the web platform.

**Generated by**: Web configurator at https://dawson.dev

**Usage**: `framework pull <token>`

**Format**: Alphanumeric string (e.g., `abc123xyz`)

**What it contains**: Reference to project config stored in Supabase

**Security**: Tokens expire after use or time limit

**Related**: Pull, Web Platform

---

### Manifest
**Noun**: JSON file tracking project metadata and framework state.

**Location**: `.dd/manifest.json` in exported projects

**Contents**:
- Framework version
- Template ID
- Applied integrations
- Export timestamp
- Project name

**Usage**: Enables drift detection, upgrade tracking, plugin context

**Related**: Export, Pull, Drift

---

### Plugin
**Noun**: User-defined code that hooks into CLI lifecycle events.

**File format**: `.mjs` (JavaScript ESM)

**Structure**:
```javascript
export default {
  name: "my-plugin",
  version: "1.0.0",
  hooks: {
    "post:export": async (context) => { /* ... */ }
  }
}
```

**Commands**:
- `framework plugin add <path>`
- `framework plugin remove <name>`
- `framework plugin list`

**Hooks Available**:
- `pre:export`, `post:export`
- `pre:deploy`, `post:deploy`
- `pre:pull`, `post:pull`

**Related**: Hook, Lifecycle Event

---

### Hook
**Noun**: Named event in the CLI lifecycle where plugins can execute.

**Format**: `<timing>:<command>` (e.g., `post:export`, `pre:deploy`)

**Timing**:
- `pre:` - Before command executes
- `post:` - After command completes

**Context**: Plugins receive context object with command data

**Related**: Plugin, Lifecycle Event

---

### Drift
**Noun**: Differences between current project state and original template.

**Command**: `framework drift`

**What it shows**:
- Files added since export
- Files deleted since export
- Files modified since export

**Use cases**:
- Understand customizations
- Prepare for template upgrades
- Document changes

**Related**: Manifest, Template

---

### Checkpoint
**Noun**: Saved snapshot of project state for rollback.

**Commands**:
- `framework checkpoint create <name>`
- `framework checkpoint restore <id>`
- `framework checkpoint list`

**What it saves**: Git commit hash or project state snapshot

**Use case**: Safety net before major refactors or agent changes

**Related**: Agent Safety, Rollback

---

### Cursor
**Noun**: AI-native code editor built on VS Code.

**Framework features**:
- `--cursor` flag generates `.cursorrules` file
- Auto-generates AI context in `prompts/` directory
- Agent safety with checkpoints

**Files generated**:
- `.cursorrules` - Editor-specific rules
- `prompts/start.md` - Starting prompt for AI
- `prompts/context.md` - Project context

**Related**: Pull, Agent, Cursorrules

---

### Cursorrules
**Noun**: Configuration file for Cursor editor behavior.

**Location**: `.cursorrules` in project root

**Format**: Markdown with embedded instructions

**Generated by**: `framework pull <token> --cursor`

**Contents**:
- Project structure context
- Code style preferences
- Common patterns
- What NOT to do

**Related**: Cursor, Agent Context

---

## Architecture Terms

### CLI
**Noun**: Command-line interface - the `framework` command you run in terminal.

**Entry point**: `bin/framework.js`

**Language**: JavaScript ESM (`.mjs` files)

**Core modules**: Located in `src/dd/`

**Related**: Core Modules, Framework

---

### Core Modules
**Noun**: Reusable logic for CLI commands, located in `src/dd/`.

**Language**: JavaScript ESM (`.mjs` files)

**Examples**:
- `src/dd/integrations.mjs` - Integration handling
- `src/dd/pull.mjs` - Pull command logic
- `src/dd/plugins.mjs` - Plugin system
- `src/dd/logger.mjs` - Logging utilities

**Naming**: `kebab-case.mjs`

**Related**: CLI, JavaScript ESM

---

### Website
**Noun**: Web-based visual configurator for building projects.

**Location**: `website/` directory

**Stack**: Next.js 15, TypeScript, React Server Components

**URL**: https://dawson.dev

**Features**:
- Template selection
- Integration configuration
- Visual preview
- Token generation for `framework pull`

**Related**: Web Platform, Pull, Token

---

### Web Platform
**Noun**: Hosted service for visual project configuration (synonymous with Website).

**Backend**: Supabase (database, auth, storage)

**Frontend**: Next.js app in `website/`

**Workflow**: Configure → Get Token → `framework pull <token>`

**Related**: Website, Token, Pull

---

### Framework Map
**Noun**: Auto-generated architecture documentation showing dependencies.

**File**: `FRAMEWORK_MAP.md`

**Generated by**: `npm run framework:map`

**Contents**:
- File dependency graph
- Import/export relationships
- Module usage

**Use case**: Understand codebase structure quickly

**Related**: Architecture, Documentation

---

### Agent Context
**Noun**: Governance file that all AI agents must read before starting work.

**File**: `AGENT_CONTEXT.md`

**Contents**:
- Project philosophy
- Verification questions
- Coding standards summary
- What NOT to do

**Required**: Every agent must answer verification questions

**Related**: Claude.md, Agent, Governance

---

### Governance
**Noun**: System of standards, rules, and documentation for consistent AI agent collaboration.

**Key Files**:
- `AGENT_CONTEXT.md` - Core context
- `CLAUDE.md` - Claude CLI auto-context
- `.cursorrules` - Cursor rules
- `docs/standards/*` - Style guides

**Purpose**: Ensure agents follow project conventions

**Related**: Agent Context, Standards

---

## Command Terms

### Deploy
**Verb**: Push project to production hosting.

**Command**: `framework deploy [--prod]`

**Providers**: Vercel, Netlify, Railway (auto-detected)

**Process**:
1. Detect provider from config
2. Validate credentials
3. Trigger deployment
4. Stream build logs
5. Return live URL

**Flags**:
- `--prod` - Deploy to production environment
- `--staging` - Deploy to staging (if available)

**Related**: Provider Detection, Credentials

---

### Doctor
**Noun**: Health check command that validates project state.

**Command**: `framework doctor [path]`

**Checks**:
- File structure integrity
- Missing dependencies
- Invalid integration configs
- Environment variable issues

**Output**: List of issues with fix suggestions

**Related**: Validation, Health Check

---

### Export (Command)
**Command**: `framework export <template-id> <output-dir> [options]`

**Purpose**: Create new project from template

**Options**:
- `--auth <provider>` - Add auth integration
- `--payments <provider>` - Add billing
- `--email <provider>` - Add email
- `--ai <provider>` - Add AI integration
- `--analytics <provider>` - Add analytics
- `--db <provider>` - Add database

**Example**:
```bash
framework export saas ./my-app --auth supabase --payments stripe
```

**Related**: Template, Integration

---

### Pull (Command)
**Command**: `framework pull <token> [options]`

**Purpose**: Download project from web platform

**Options**:
- `--cursor` - Generate Cursor-specific files
- `--open` - Open in Cursor after pull

**Example**:
```bash
framework pull abc123xyz --cursor --open
```

**Related**: Token, Cursor

---

### Templates (Command)
**Command**: `framework templates <subcommand>`

**Subcommands**:
- `list` - Show all available templates
- `search <query>` - Find templates by keyword
- `info <template-id>` - Show template details

**Example**:
```bash
framework templates list
framework templates info saas
```

**Related**: Template, Template ID

---

### Plugin (Command)
**Command**: `framework plugin <subcommand>`

**Subcommands**:
- `add <path>` - Install a plugin
- `remove <name>` - Uninstall a plugin
- `list` - Show installed plugins

**Example**:
```bash
framework plugin add ./my-plugin.mjs
framework plugin list
```

**Related**: Plugin, Hook

## File Types

### JavaScript ESM
**Format**: ES Modules with `.mjs` extension

**Where used**: CLI and core modules (`bin/`, `src/dd/`, `tests/`)

**Style**:
- No semicolons
- 2-space indent
- Import Node.js built-ins with `node:` prefix

**Example**:
```javascript
import fs from "node:fs"
import path from "node:path"

export function doSomething() {
  // No semicolons
  return true
}
```

**Related**: Core Modules, CLI, Coding Standards

---

### TypeScript
**Format**: `.ts` and `.tsx` files

**Where used**: Website (`website/`), templates (`templates/`)

**Style**:
- Semicolons required
- 2-space indent
- Explicit types
- Path aliases (`@/components`)

**Example**:
```typescript
import { useState } from "react";

export function MyComponent(): JSX.Element {
  const [value, setValue] = useState<string>("");
  return <div>{value}</div>;
}
```

**Related**: Website, Templates, Coding Standards

---

### Integration JSON
**File**: `integration.json`

**Location**: `templates/{template}/integrations/{type}/{provider}/integration.json`

**Purpose**: Metadata describing an integration

**Required fields**:
- `name` - Human-readable name
- `type` - Integration type (auth, payments, etc.)
- `provider` - Provider name
- `version` - Integration version
- `dependencies` - npm packages to install

**Example**:
```json
{
  "name": "Supabase Auth",
  "type": "auth",
  "provider": "supabase",
  "version": "1.0.0",
  "dependencies": {
    "@supabase/supabase-js": "^2.47.0"
  }
}
```

**Related**: Integration, Provider

---

### Template JSON
**File**: `template.json`

**Location**: `templates/{template-id}/template.json`

**Purpose**: Metadata describing a template

**Required fields**:
- `id` - Template ID
- `name` - Human-readable name
- `version` - Template version
- `description` - Brief description
- `supportedIntegrations` - Compatible integrations

**Example**: See `templates/saas/template.json`

**Related**: Template, Template ID

---

### Manifest JSON
**File**: `manifest.json`

**Location**: `.dd/manifest.json` in exported projects

**Purpose**: Track project state and metadata

**Contents**:
- `frameworkVersion` - Version used for export
- `templateId` - Source template
- `integrations` - Applied integrations
- `createdAt` - Timestamp

**Generated by**: Export and Pull commands

**Related**: Manifest, Export, Pull

## Development Terms

### Agent
**Noun**: AI assistant (like Claude) working on the codebase.

**Governance**: Must read `AGENT_CONTEXT.md` before starting work

**Safety features**:
- Checkpoints for rollback
- Verification questions
- Mandatory reading requirements

**Related**: Agent Context, Governance, Cursor

---

### Agent Safety
**Noun**: Features preventing AI agents from breaking projects.

**Features**:
- Checkpoints (rollback points)
- Audit log (track agent changes)
- Drift detection (see what changed)
- Verification tests (ensure understanding)

**Commands**: `framework checkpoint`, `framework drift`

**Related**: Agent, Checkpoint, Drift

---

### Lifecycle Event
**Noun**: Named point in CLI execution where plugins can hook.

**Format**: `<timing>:<command>` (e.g., `post:export`)

**Available events**:
- `pre:export`, `post:export`
- `pre:deploy`, `post:deploy`
- `pre:pull`, `post:pull`

**Related**: Plugin, Hook

---

### Provider Detection
**Noun**: Auto-detecting which deployment platform to use.

**Command**: `framework deploy` (no flags needed)

**Detection order**:
1. Check for `vercel.json` → Use Vercel
2. Check for `netlify.toml` → Use Netlify
3. Check for `railway.json` → Use Railway
4. Ask user to choose

**Related**: Deploy, Vercel, Netlify

---

### Recovery Guidance
**Noun**: Actionable error messages that tell you how to fix issues.

**Example**:
```
Error: Config file not found: .dd/config.json
Fix: Run 'framework init' to create a config file
```

**Implementation**: `src/dd/recovery-guidance.mjs`

**Philosophy**: Never show errors without solutions

**Related**: Error Handling, Logger

## Relationships

### Template → Integration
Templates declare which integrations they support in `template.json`:

```json
{
  "supportedIntegrations": {
    "auth": ["supabase", "clerk"],
    "payments": ["stripe", "paddle"]
  }
}
```

### Integration → Provider
An integration is the combination of type + provider:
- Type: `auth`
- Provider: `supabase`
- Full integration: `auth/supabase`

### Export → Pull
Both create projects, but from different sources:
- **Export**: From local/remote template
- **Pull**: From web platform configuration

### Plugin → Hook
Plugins register functions to run at hooks:
- Plugin: User code
- Hook: Named lifecycle event
- Result: Plugin executes when hook fires
